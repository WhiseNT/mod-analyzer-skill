# 模组设计分析报告模板

> 本模板从游戏设计角度评价一个 mod 的体验质量。使用前必须先完成"游戏内容文档"中的基础探查（阶段 0~3），再基于证据做设计判断。不要保留无证据的断言。

# 《[Mod 名称]》模组设计分析报告

## 0. 一句话结论

用 2-4 句话说明：这个 mod 的设计水平如何、核心设计优点是什么、最大设计问题是什么、适合什么类型的整合包场景。

- 配置项默认值是否反映了作者预期的体验方向？

## 2. 核心玩法循环评价

### 2.1 循环完整性

```text
[触发/入口] → [资源获取] → [加工/交互] → [风险/限制] → [奖励/产出] → [解锁/升级] → [回到下一轮]
```

| 环节 | 是否存在 | 设计质量 | 问题 |
|------|----------|----------|------|
| 触发/入口 | 是/否/弱 | 高/中/低 | |
| 资源获取 | 是/否/弱 | 高/中/低 | |
| 加工/交互 | 是/否/弱 | 高/中/低 | |
| 风险/限制 | 是/否/弱 | 高/中/低 | |
| 奖励/产出 | 是/否/弱 | 高/中/低 | |
| 解锁/升级 | 是/否/弱 | 高/中/低 | |
| 回流 | 是/否/弱 | 高/中/低 | |

### 2.2 循环强度判断

- 循环是否闭合？是否存在断裂环节？
- 玩家是否会自然进入循环，还是需要外部引导？
- 循环是否有明确的反馈机制和升级路径？
- 是否存在"为了复杂而复杂"的冗余环节？

### 2.3 循环类型分类

- **强循环**：玩家反复执行同一流程，每次有明确反馈和成长（科技线、战斗线）
- **弱循环**：玩家偶尔使用，没有持续的重复动机（工具型、装饰型）
- **渐进循环**：每次循环解锁新内容，循环本身在演变（冒险线、进度线）
- **单次循环**：玩家完成一次后没有重复动机（剧情线、探索线）

### 3.2 首次接触体验

- 玩家第一次遇到这个 mod 的内容是在什么情境下？
- 玩家能否自然理解"这个东西怎么用"？
- 是否存在"不查 wiki 就无法上手"的黑箱机制？
- 关键机制的首次使用是否有明确的反馈？

### 3.3 学习曲线判断

- 复杂度递进是否平滑？是否存在突然的难度/复杂度跳跃？
- 新机制引入时是否有足够的上下文让玩家理解？
- 是否存在"学了前面才能理解后面"的正确依赖顺序？

### 4.2 阶段划分评价

- 阶段划分是否清晰？玩家能否感知"我进入了新阶段"？
- 阶段之间的过渡是否有明确的里程碑/标志物？
- 是否存在"跨阶段内容"（过早可获取的后期内容，或过晚解锁的初期内容）？

### 4.3 内容密度评价

- 内容量是否与阶段匹配？是否存在某阶段内容过少（空虚）或过多（拥挤）？
- 物品/方块/实体的数量是否在合理范围内？
- 是否存在大量"凑数内容"（功能重复、用途不明的物品）？

### 5.2 风险/惩罚分析

- 玩家失败的成本是什么？（物品损失、时间损失、进度回退）
- 失败条件是否对玩家透明？是否有提前预警？
- 是否存在"非预期死亡导致进度严重回退"的设计？
- 风险是否与回报匹配？

### 5.3 重复劳动检测

- 是否存在大量"等进度"（现实时间等待、固定 tick 间隔）？
- 是否存在"为了延长游戏时间而设计"的重复操作？
- 是否存在"不自动化就无法推进"的门槛？
- 配置项中是否有调整重复劳动强度的参数？
